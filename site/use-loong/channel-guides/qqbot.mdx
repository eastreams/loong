# QQ Bot

This is a shipped plugin-backed bridge surface. Loong owns the surface id, config contract, and discovery UX, while the active QQ gateway or bridge owns the upstream runtime.

## At A Glance

| Field | Value |
| --- | --- |
| Catalog id | `qqbot` |
| Config key | `qqbot` |
| Implementation status | `plugin_backed` |
| Transport | `qq_official_bot_gateway_or_plugin_bridge` |
| Aliases | `qq`, `qq-bot`, `tencent-qq` |
| Default send target kind | `conversation` |
| Runtime owner | external plugin |

## Minimal Config

```toml
[qqbot]
enabled = true
app_id_env = "QQBOT_APP_ID"
client_secret_env = "QQBOT_CLIENT_SECRET"
allowed_peer_ids = ["openid_alice"]
```

When several compatible managed bridges are installed, pin the one Loong should trust:

```toml
[qqbot]
enabled = true
managed_bridge_plugin_id = "qqbot-bridge-guided"
app_id_env = "QQBOT_APP_ID"
client_secret_env = "QQBOT_CLIENT_SECRET"
allowed_peer_ids = ["openid_alice"]
```

## Smoke Test

```bash
loong channels
loong channels --json
loong doctor
```

What success looks like:

- `QQ Bot [qqbot]` appears in the plugin-backed section
- managed bridge discovery shows one compatible plugin or a concrete ambiguity/remediation detail
- doctor keeps bridge-send / bridge-serve contract checks visible and actionable

## Stable Target Templates

| Target | Meaning |
| --- | --- |
| `qqbot:<account>:c2c:<openid>` | direct message openid |
| `qqbot:<account>:group:<openid>` | group openid |
| `qqbot:<account>:channel:<id>` | guild channel id |

## Account Scope Note

QQ Bot openids are scoped to the selected account. Keep account ids stable so route meaning does not drift when you rotate gateway credentials or bridge implementations.

## Managed Bridge Discovery Notes

- Compatible managed bridges can advertise required config keys and setup docs URLs that Loong then surfaces through `doctor`.
- If discovery finds several compatible bridges, `managed_bridge_plugin_id` is the correct tie-breaker.
- If bridge metadata is incomplete, Loong keeps the surface visible and reports the missing fields instead of flattening everything into one generic placeholder.

## Gateway And Ownership

This surface does **not** join `multi-channel-serve` or `gateway run` today. The official QQ gateway or compatible bridge remains the runtime owner.

## Operator Notes

- Keep `allowed_peer_ids` explicit and narrow.
- Treat QQ Bot as a multi-account surface from the start if you expect separate direct, group, or guild identities.
- Prefer one selected managed bridge per environment rather than leaving ambiguity unresolved.
